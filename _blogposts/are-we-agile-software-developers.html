A brief history of Agile

Agile software development is a phrase coined in 2001 by a group of software developers who got together in Utah and wanted to change the industry. Agile was essentially born out of a frustration for the way software development was traditionally done. It used to take a long time to develop software. It could take years before a project was actually finished and in that time the client couldn’t use the software. Software wasn’t prioritized and the development of it was only started after the hardware was completed. Once it had been built it was very difficult to change it, and with a lack of feedback it was also unclear whether or not the software would actually work well for the client.

Essentially, software development used to be a clunky process that led to bad software being developed. Something needed to change and the agile manifesto was born (have a look at agile­manifesto.org for the original website). The agile manifesto is a collection of principles and values for software developers to follow in order to deliver and create better software. The agile values are:

Individuals and interactions over processes and tools
Working software over comprehensive documentation
Customer collaboration over contract negotiations
Responding to change over following a plan
All of the items on the list are valued, but the ones on the left are valued more than those on the right.

What is Agile (not)?

One of the most important things to note about agile is that it is a set of values and principles. Agile is therefore not a decision maker; a methodology to follow; a specific way of developing software; a framework to follow or scrum.

All software development companies today will say that they are agile. Most of them will make this claim because they use scrum, a methodology created based on the agile principles. Scrum orders the workflow into sprints where small bits of the software is planned, built, tested and reviewed and the process is repeated as many times as is needed for the complete software to be produced.

The problem with scrum is that companies often use it without considering the agile values and principles. They are using a tool blindly and end up following the scrum process religiously which then ends up valuing processes and tools over individuals and interactions, going against the agile values listed above.

{% capture imageDescription %} Scrumfall

Scrumfall combines waterfall and scrum. Waterfall is the traditional method of producing code, where research and documentation is completed before any code is created. Many big companies have used this methods, but have tried to modernize by including scrum within that process, to make them seem agile. This usually leads to scrumfall, where the two processes fail to combine because they work in very different ways, so the company is really still using the traditional waterfall method with requirement specifications and no continual deliveries, but can say they're agile because they've implemented scrum within the waterfall process.

“The problem with hearing the word agile in the software industry today, and in particular in Denmark, is that people equate agile to scrum. And although scrum was initially based on the agile principles, the process of using scrum itself isn’t actually agile. That’s why we, at deranged, don’t say that we’re agile, despite the fact that we use the agile principles and values every day, because we don’t want people to think we use scrum. We don’t follow a particular set methodology, like scrum, because our clients and their projects are different and require different approaches,” explains Niels from deranged.

Agile is also not a case of having a huge requirement specification (comprehensive documentation) before being able to create working software, or having a contract in place before you can start to consider working with another company, and it certainly doesn’t mean sticking to a plan at any cost if other more important things come up. As the name suggests, agile requires agility from developers.

Agile software development at deranged

We have to start by pointing out that we don’t love saying that we are agile developers simply because it has become so tainted. We don’t use scrum or any other agile methodology, but we do follow the agile principles and values, mainly because we find it natural to do so with the way we work.

We consider each of our clients to be individuals. Every single one will require something slightly different from us, and we need to be able to provide them with that. We don’t follow a set structure with each of our clients, but find a way to work together that suits both us and them. That way we can make sure that we’re prioritising the people we’re working with, establish a good customer collaboration and provide better work for them because we have a better understanding of how they work.

For us, it's not important that we're agile. what's important is that we work with our clients to understand what their needs are, deliver software continually and that we don't get lost in documentation and processes. Most importantly we want to be fair and honest developers who say no when we need to, find the best solutions for a client and think both in the long and the short term.

We also never work with requirement specifications. We will establish what is needed and what will work best alongside our client and throughout the process. We won’t ask the clients to figure it out on their own. That ultimately makes for counterproductive work, as it is impossible to predict everything that is going to be needed for the software development.

When creating our fee for any project, we set out three different price estimates in order to allow for any unforeseen changes along the way. We strongly believe that it is cheaper in the long run to fix things as they come up, rather than to push on and create software that isn’t optimal for the client. We also believe in providing our clients with changes quickly and efficiently, so we frequently deliver working software to our clients, so they can see how those developments help to build their business.

We use the agile principles and values to help us to deliver the best software solutions to our clients. However, due to the fact that agile has become such a loaded term, we tend to not market ourselves on it. It is innate to the way we work, that we value our clients over a process, that we’d rather create working software than wait for a requirement specification, that we’d rather work with our clients instead of negotiate with them and that we’d rather create software that actually works instead of stick to a precise plan.

If a software developer tells you they’re agile, ask them how they do it, which values and principles they find particularly interesting or have a specific focus on. Agile is not just a word or something to market yourself on – find out how they’re actually using it within the company. The values and processes behind agile software development should make the process better for you as a client, you just have to find out whether or not the developers know why and how that is.
